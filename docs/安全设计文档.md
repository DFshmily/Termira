# Termira 安全设计文档

## 1. 文档目标

本文档定义 Termira 第一阶段和第二阶段的安全设计边界，重点说明本地 Vault、可选主密码、敏感信息存储、日志脱敏、第一阶段不实现的 Keychain 和 known_hosts 能力，以及第二阶段迁移方案。

第一阶段安全目标：

- 不明文保存密码。
- 不明文保存私钥 passphrase。
- 不明文保存可选私钥内容。
- 日志不输出敏感信息。
- 提供接近 FinalShell 本地保存体验。
- 主密码可选。

第一阶段非目标：

- 不提供 Keychain 级别保护。
- 不实现 known_hosts。
- 不实现首次指纹确认。
- 不实现指纹变更阻断。
- 不实现云端端到端加密同步。

## 2. 安全模型

第一阶段安全模型分两种模式。

### 2.1 无主密码模式

用户体验：

- 类似传统本地 SSH 客户端。
- 保存凭据后可直接连接。
- 用户无需每次输入主密码。

安全特征：

- 凭据不明文落盘。
- 使用本地应用加密材料保护 Vault。
- 可防止普通误查看、普通日志泄露和配置文件直接泄露。

安全边界：

- 如果攻击者获得完整本机文件访问权限，并能读取应用文件和 Vault 文件，则防护弱于 Keychain。
- 不应宣传为零知识加密。
- 不应宣传为系统级凭据保护。

### 2.2 可选主密码模式

用户体验：

- 用户可以设置主密码。
- 启动后或 Vault 锁定后需要解锁。
- 解锁后可以使用保存的凭据连接。

安全特征：

- Vault key 由主密码派生。
- 忘记主密码无法恢复 Vault 明文。
- 安全性高于无主密码模式。

安全边界：

- 主密码强度决定离线攻击难度。
- 需要合理 KDF 参数。
- 不应在任何日志中记录主密码。

## 3. Vault 设计

### 3.1 Vault 保存内容

Vault 保存敏感信息：

- SSH password。
- private key passphrase。
- 可选 private key content。

Vault 不保存非敏感连接信息：

- host。
- port。
- username。
- display name。
- group。
- tags。
- note。

非敏感连接信息由 profile store 保存。

### 3.2 CredentialRef

profile store 只保存 `credentialRef`。

示例：

```json
{
  "id": "host_001",
  "name": "Production API",
  "host": "10.0.0.10",
  "port": 22,
  "username": "deploy",
  "auth": {
    "type": "password",
    "credentialRef": "cred_001",
    "saveCredential": true
  }
}
```

Vault 内部保存：

```json
{
  "credentialId": "cred_001",
  "type": "password",
  "ciphertext": "...",
  "createdAt": "...",
  "updatedAt": "..."
}
```

### 3.3 Vault 文件结构

建议结构：

```json
{
  "schemaVersion": 1,
  "kdf": {
    "name": "argon2id",
    "params": {
      "memoryKiB": 65536,
      "iterations": 3,
      "parallelism": 1
    },
    "salt": "base64"
  },
  "cipher": {
    "name": "xchacha20-poly1305",
    "nonce": "base64"
  },
  "mode": "master-password",
  "payload": "base64"
}
```

如果 Java 第一阶段依赖实现限制，也可以使用：

- PBKDF2WithHmacSHA256 + AES-256-GCM。

但推荐优先评估 BouncyCastle 或 libsodium Java binding，以便更接近现代 Vault 设计。

### 3.4 加密算法建议

优先方案：

- KDF：Argon2id。
- AEAD：XChaCha20-Poly1305。

备选方案：

- KDF：PBKDF2WithHmacSHA256。
- AEAD：AES-256-GCM。

选择规则：

- 如果依赖成熟、打包稳定，优先 Argon2id + XChaCha20-Poly1305。
- 如果 Java 打包和跨平台稳定性优先，第一阶段可使用 PBKDF2 + AES-GCM，但必须记录 KDF 参数并预留迁移。

## 4. 主密码策略

主密码是可选功能。

设置主密码时：

- 不保存主密码。
- 只保存 salt、KDF 参数和加密 payload。
- 解锁时用用户输入派生 key。
- 解锁失败返回统一错误。

不设置主密码时：

- 应用生成本地加密 key。
- key 不能硬编码在源码中。
- key 不能写入日志。
- key 文件权限尽量收紧。
- 文档和 UI 应提示安全级别低于主密码和 Keychain。

## 5. 日志脱敏

禁止记录：

- password。
- passphrase。
- private key。
- master password。
- Vault 明文。
- 终端输入。
- 终端完整输出。

允许记录：

- profileId。
- sessionId。
- credentialRef。
- host 脱敏。
- port。
- 错误码。
- 调用耗时。
- 连接状态。

示例：

```text
INFO ssh.connect profileId=host_001 host=10.0.*.* port=22 status=connecting
WARN ssh.connect profileId=host_001 error=SSH_AUTH_FAILED elapsedMs=1200
```

## 6. 第一阶段不实现 known_hosts 的风险

第一阶段明确不实现：

- known_hosts。
- 首次指纹确认。
- 指纹变更阻断。

风险：

- 用户无法在第一阶段通过产品确认远端主机身份。
- 无法自动阻断中间人攻击场景。
- 安全性低于完整 SSH 客户端最佳实践。

缓解：

- 在文档中明确。
- 在 UI 的高级安全说明中标注第一阶段限制。
- 第二阶段优先补齐。

第一阶段不得在 UI 或文档中宣称“完整主机身份校验”。

## 7. 第二阶段 Keychain 方案

第二阶段接入：

- macOS Keychain。
- Windows Credential Manager。

推荐模型：

- Vault 文件仍保留。
- 本地随机解锁 key 由系统凭据管理器保护。
- 用户平时无感打开。
- 高安全模式可继续要求主密码。

macOS：

```text
Termira local vault key -> macOS Keychain
Encrypted vault payload -> local file
```

Windows：

```text
Termira local vault key -> Windows Credential Manager
Encrypted vault payload -> local file
```

## 8. 第二阶段 known_hosts 方案

第二阶段实现：

- 首次连接显示主机指纹。
- 用户确认后保存 known_hosts。
- 后续连接自动校验。
- 指纹变更时阻断连接。
- 用户必须明确选择更新指纹。

known_hosts 字段：

- host。
- port。
- algorithm。
- fingerprint。
- firstSeenAt。
- lastSeenAt。

指纹变更行为：

- 默认阻断。
- 展示旧指纹和新指纹。
- 提示安全风险。
- 只有用户明确确认后更新。

## 9. Vault 迁移策略

Vault 必须包含：

- schemaVersion。
- cipher name。
- kdf name。
- kdf params。

迁移原则：

- 新版本读取旧 Vault。
- 迁移前备份旧 Vault。
- 迁移失败不覆盖旧 Vault。
- 迁移成功后写入新 schema。

第一阶段到第二阶段迁移：

1. 读取第一阶段 Vault。
2. 用户解锁。
3. 生成系统凭据管理器保护的随机 key。
4. 使用新 key 重加密 Vault。
5. 写入新 schema。
6. 保留备份。

## 10. UI 安全提示

第一阶段应在设置中展示安全模式：

- 本地 Vault。
- 主密码：未启用/已启用。
- Keychain：第二阶段支持。
- 主机指纹校验：第二阶段支持。

文案原则：

- 真实说明安全边界。
- 不制造恐慌。
- 不夸大保护能力。

建议提示：

```text
当前版本使用本地加密 Vault 保存敏感凭据。启用主密码可提高离线保护强度。系统 Keychain 与主机指纹校验将在后续版本提供。
```

## 11. 安全验收清单

第一阶段必须通过：

- 配置库中无明文 password。
- 配置库中无明文 passphrase。
- 配置库中无明文 private key content。
- 日志中无敏感凭据。
- 主密码错误不能解锁 Vault。
- Vault 损坏不覆盖原文件。
- 应用崩溃后 Vault 不产生半写入。

第二阶段必须通过：

- Keychain 保存本地 key。
- 删除 Keychain 项后无法自动解锁。
- known_hosts 首次确认生效。
- 指纹变更默认阻断。
- Vault 迁移失败可回滚。
