# Termira 架构文档

## 1. 架构目标

Termira 第一阶段采用 Electron + React + Java sidecar 的架构，目标是在保证开发可控性的前提下实现完整 SSH 客户端核心能力。

架构目标：

- UI 与 SSH 内核解耦。
- IPC 协议稳定。
- Java 内核可由当前团队维护。
- 未来可替换 Rust 内核。
- 支持 macOS 第一阶段交付。
- 支持后续 Windows 扩展。

## 2. 总体架构

```text
+----------------------------------------------------+
|                    Electron App                    |
|                                                    |
|  +-------------------+   +----------------------+  |
|  | React Renderer    |   | Electron Main        |  |
|  |                   |   |                      |  |
|  | Workbench UI      |   | Window/Menu          |  |
|  | xterm.js          |   | IPC Bridge           |  |
|  | SFTP Panel        |   | Sidecar Lifecycle    |  |
|  | Forwarding Panel  |   | App Lifecycle        |  |
|  +---------+---------+   +----------+-----------+  |
|            |                        |              |
|            | preload API            | stdio RPC    |
|            v                        v              |
|  +------------------------------------------------+ |
|  |              Java SSH Sidecar                  | |
|  | SSH / SFTP / Forwarding / Monitor / Vault      | |
|  +------------------------------------------------+ |
+----------------------------------------------------+
                         |
                         v
              Remote SSH Servers / Local Files
```

## 3. 分层职责

### 3.1 React Renderer

负责：

- 工作台 UI。
- 主机树。
- 终端标签。
- xterm 渲染。
- SFTP 文件面板。
- 端口转发面板。
- 监控面板。
- 进程面板。
- 快捷命令面板。
- 用户设置。
- UI 语言资源选择，默认中文，保留英文。
- 后端运行状态展示与基础诊断入口。

不负责：

- 直接读取本地敏感文件。
- 直接访问 Node API。
- 直接连接 SSH。
- 直接解密 Vault。

### 3.2 Electron Preload

负责：

- 暴露安全 API 给 renderer。
- 屏蔽 Node 能力。
- 提供 `invoke` 和事件订阅。

### 3.3 Electron Main

负责：

- 创建窗口。
- 管理菜单。
- 管理应用生命周期。
- 启动 Java sidecar。
- 停止 Java sidecar。
- 转发 IPC 请求。
- 处理 sidecar 退出。
- 将 sidecar 事件广播给 renderer。

Electron main 不实现 SSH 业务逻辑。

### 3.4 Java Sidecar

负责：

- SSH 连接。
- 终端通道。
- SFTP。
- 端口转发。
- 服务器监控。
- 进程管理。
- 主机配置存储。
- Vault 加密凭据存储。
- 统一错误码。
- 结构化日志。

## 4. 进程模型

第一阶段进程：

- Electron main process。
- Electron renderer process。
- Java sidecar process。

Java sidecar 生命周期：

1. Electron main 启动。
2. Electron main spawn Java sidecar。
3. Java sidecar 初始化。
4. Java sidecar 发出 `backend.ready`。
5. Renderer 可以发起业务请求。
6. App 退出时 Electron main 发送 `app.shutdown`。
7. Java sidecar 关闭所有 SSH session、SFTP session、forwarding。
8. Java sidecar 退出。

异常处理：

- sidecar 非预期退出时，Electron main 发送 `backend.exited`。
- Renderer 在设置的运行状态中展示后端不可用状态。
- 用户可以在设置中触发重启 sidecar。

UI 状态原则：

- 产品首屏保持工作台，不展示工程基线或后端调试页。
- 后端生命周期、Ping、重启、停止等诊断能力归入设置。
- 后端不可用不应导致主机列表和静态工作台崩溃。

## 5. IPC 架构

### 5.1 通信协议

第一阶段使用 stdio JSON-RPC 风格协议。

请求：

```json
{
  "id": "req_1001",
  "type": "request",
  "method": "terminal.write",
  "params": {
    "sessionId": "ses_1",
    "channelId": "ch_1",
    "data": "ls\n"
  }
}
```

响应：

```json
{
  "id": "req_1001",
  "type": "response",
  "ok": true,
  "result": {}
}
```

事件：

```json
{
  "type": "event",
  "event": "terminal.output",
  "payload": {
    "sessionId": "ses_1",
    "channelId": "ch_1",
    "data": "README.md\r\n"
  }
}
```

### 5.2 协议版本

每次连接 sidecar 时，Electron main 应调用：

- `app.getVersion`

返回：

```json
{
  "protocolVersion": "1.0.0",
  "backendVersion": "0.1.0"
}
```

协议不兼容时：

- 阻止继续调用。
- 展示明确错误。
- 提示更新应用。

## 7. SSH 模块架构

```text
SshSessionManager
  -> SshAuthService
  -> SshSession
      -> TerminalChannel
      -> ExecChannel
      -> SftpSession
      -> ForwardingHandle
```

### 7.1 SshSessionManager

职责：

- 创建连接。
- 认证。
- 保存连接状态。
- 提供 session 查询。
- 关闭 session。
- 派发状态事件。

### 7.2 SshAuthService

职责：

- 从 Vault 获取凭据。
- 组合认证参数。
- 执行 password/private key/keyboard-interactive。
- 返回认证结果。

### 7.3 TerminalChannel

职责：

- 请求 PTY。
- 打开 shell。
- 读写数据。
- resize。
- 关闭。

### 7.4 ExecChannel

职责：

- 执行监控命令。
- 执行进程命令。
- 执行一次性快捷命令。

## 8. SFTP 模块架构

```text
SftpManager
  -> SftpSession
  -> TransferManager
      -> UploadTask
      -> DownloadTask
```

SFTP 需要复用已连接 SSH session。每个 session 可以按需创建 SFTP client。

传输任务事件：

- `transfer.progress`
- `transfer.completed`
- `transfer.failed`

## 9. 端口转发模块架构

```text
ForwardingManager
  -> LocalForwardingService
  -> RemoteForwardingService
  -> DynamicSocksService
```

Local forwarding：

```text
localhost:localPort -> SSH tunnel -> targetHost:targetPort
```

Remote forwarding：

```text
remoteBindHost:remoteBindPort -> SSH tunnel -> local target
```

Dynamic forwarding：

```text
localhost:socksPort -> SOCKS proxy -> SSH tunnel -> remote targets
```

## 10. 监控模块架构

```text
MonitorManager
  -> MonitorScheduler
  -> LinuxMonitorCollector
  -> Parser
```

监控采集必须：

- 使用独立调度。
- 设置超时。
- 捕获错误。
- 不影响终端通道。

## 11. 存储架构

### 11.1 配置存储

推荐 SQLite：

```text
profiles.db
```

保存：

- 主机配置。
- 主机分组。
- 端口转发规则。
- 快捷命令。
- 最近连接。
- UI 偏好。

### 11.2 Vault 存储

推荐独立文件：

```text
vault.dat
```

保存：

- 加密后的凭据。
- salt。
- nonce。
- schema version。
- KDF 参数。

不保存明文敏感信息。

## 12. Rust 预留方案

Rust 不是第一阶段实现目标，但架构必须预留。

预留原则：

- Renderer 只依赖 IPC 方法，不依赖 Java 类。
- Electron main 只管理 sidecar，不关心 sidecar 内部语言。
- Java sidecar 与未来 Rust sidecar 使用同一协议版本策略。
- 数据 schema 保持语言无关。

未来替换方式：

1. 实现 Rust sidecar。
2. 支持相同 IPC 方法。
3. 支持相同事件格式。
4. 通过配置选择 sidecar。
5. 逐步替换模块。

## 13. 第一阶段明确不实现的架构能力

第一阶段不实现：

- Keychain adapter。
- Windows Credential Manager adapter。
- known_hosts store。
- Host key verification flow。
- 云同步服务。
- 团队空间。
- 审计服务。

但相关接口命名应避免未来冲突。
