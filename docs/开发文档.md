# Termira 开发文档

## 1. 文档目标

本文档面向 Termira 后续开发者，描述第一阶段推荐技术栈、项目结构、开发环境、启动流程、IPC 协议、前端规范、Java sidecar 规范、日志规范、错误规范和常见调试方法。

第一阶段只沉淀文档时不创建项目骨架。后续开发应按本文档搭建工程。

## 2. 技术路线

第一阶段采用：

- 桌面壳：Electron。
- 前端：React + TypeScript。
- 终端组件：xterm.js。
- 样式：CSS Modules 或 Tailwind CSS 二选一，推荐先用 CSS Modules + CSS variables，避免早期引入过多抽象。
- 图标：lucide-react。
- 后端内核：Java sidecar。
- SSH 客户端库：SSHJ 优先。
- 备选 SSH 库：Apache MINA SSHD。
- 前后端通信：Electron main 管理 Java sidecar 进程，通过 stdio JSON-RPC 或 localhost loopback HTTP/WebSocket 通信。
- 本地数据：SQLite 或 JSON 文件。第一阶段推荐 SQLite 保存结构化配置，Vault 文件保存加密凭据。
- Java 构建工具：Maven。

不采用 Tauri 作为第一阶段主桌面壳，原因：

- 第一阶段后端主力是 Java，Electron 与 Java sidecar 的进程模型更直接。
- 用户当前更熟悉 Java，长期维护成本更低。
- Termius 桌面端公开资料也显示其使用 Electron，Electron 更贴近目标产品形态。

Rust 预留：

- 不在第一阶段实现 Rust SSH 内核。
- IPC 协议保持语言无关。
- 后续若替换 Java sidecar 为 Rust sidecar，前端和 Electron main 不应大规模改动。

## 3. 分阶段开发路线

Termira 必须按阶段推进。每个阶段都要能单独验收，不能把 UI、SSH 内核、SFTP、端口转发、Vault 一次性混做。后续开发时，只有当前阶段验收通过，才进入下一阶段。

### 3.1 阶段 0：工程骨架与开发基线

目标：

- 创建 monorepo。
- 建立 Electron + React + TypeScript 桌面前端。
- 建立 Maven Java sidecar。
- 打通 Electron main 启动 Java sidecar 的最小链路。

交付物：

- `apps/desktop` 前端工程。
- `apps/backend-java/pom.xml`。
- Electron main、preload、renderer 基础结构。
- Java `Main` 入口。
- `app.ping` IPC 方法。
- `backend.ready` 事件。
- 基础日志目录。

开发顺序：

1. 创建前端 Vite + React + TypeScript 工程。
2. 创建 Electron main 和 preload。
3. 创建 Maven Java 工程。
4. 实现 Java sidecar 从 stdin 读取 JSON、向 stdout 输出 JSON。
5. Electron main spawn Java sidecar。
6. Renderer 调用 `app.ping` 并展示后端在线状态。

验收标准：

- `npm run dev` 能打开桌面窗口。
- Electron main 能启动 Java sidecar。
- Renderer 能调用 `app.ping`。
- Java sidecar 退出后 UI 能显示后端离线。

- 先完成 Termius 风格的工作台 UI。

- 主机搜索。

- 中央终端区域。

- 右侧或底部工具区。
- SFTP、端口转发、监控、进程、快捷命令面板静态态。
- 深色主题 token。

开发顺序：

1. 定义 CSS variables 和基础布局。
2. 实现 AppShell。
3. 实现 HostSidebar。
4. 实现 TerminalTabs 和 TerminalPane 静态态。
5. 实现右侧工具面板容器。
6. 实现各功能面板空状态和 mock 状态。
7. 补齐 hover、selected、focus、disabled、loading、error 状态。

验收标准：

- 启动直接进入工作台。
- 不出现营销首页。
- 主机、终端、文件、端口转发入口都在首屏可见。
- 长主机名、长路径、多标签不破坏布局。
- UI 风格深色、克制、专业、高效。

开发顺序：

1. 设计 SQLite schema。
2. 实现 profile CRUD。
3. 实现 credentialRef 模型。
4. 实现 Vault 文件格式。
5. 实现无主密码模式。

- 设置主密码后，错误主密码不能解锁 Vault。

## 6. 依赖选择

### 6.1 前端依赖

推荐：

- `electron`
- `vite`
- `react`
- `react-dom`
- `typescript`
- `@xterm/xterm`
- `@xterm/addon-fit`
- `@xterm/addon-web-links`
- `lucide-react`
- `zustand` 或 `redux-toolkit`
- `zod`

状态管理推荐：

- 第一阶段优先 `zustand`。
- 原因是 SSH 客户端状态多为本地运行态，`zustand` 简洁、样板少。
- 如果后续需要复杂可回放事件流，再评估 `redux-toolkit`。

### 6.2 Java 依赖

推荐 SSHJ：

- `com.hierynomus:sshj`

推荐原因：

- 更贴近 SSH 客户端场景。
- 支持密码认证。
- 支持公钥认证。
- 支持 keyboard-interactive。
- 支持 shell。
- 支持 exec。
- 支持 SFTP。
- 支持 local/remote port forwarding。

推荐辅助依赖：

- Jackson：JSON 序列化。
- SLF4J + Logback：日志。
- SQLite JDBC：本地配置库。
- BouncyCastle：加密能力补充。
- JUnit 5：测试。
- AssertJ：测试断言。

备选 Apache MINA SSHD：

- 当需要更完整的 SSH 框架能力、服务端能力或更细粒度协议控制时评估。
- 第一阶段不建议默认使用，避免复杂度过高。

## 7. 启动与开发流程

### 7.1 前端开发流程

建议命令：

```bash
npm install
npm run dev
```

`npm run dev` 应同时启动：

- Vite dev server。
- Electron main process。
- Java sidecar。

如果拆分命令：

```bash
npm run dev:renderer
npm run dev:electron
npm run dev:java
```

### 7.2 Java sidecar Maven 命令

Java 后端统一使用 Maven，不再使用 Gradle。

推荐命令：

```bash
mvn -f apps/backend-java/pom.xml clean test
mvn -f apps/backend-java/pom.xml package
mvn -f apps/backend-java/pom.xml exec:java
```

如果使用 Maven Wrapper，命令对应为：

```bash
./mvnw -f apps/backend-java/pom.xml clean test
./mvnw -f apps/backend-java/pom.xml package
./mvnw -f apps/backend-java/pom.xml exec:java
```

`npm run dev:java` 应封装 Maven 启动命令，推荐映射为：

```bash
mvn -f apps/backend-java/pom.xml exec:java
```

`npm run build:java` 应封装 Maven 打包命令，推荐映射为：

```bash
mvn -f apps/backend-java/pom.xml clean package
```

Maven 打包产物建议：

```text
apps/backend-java/target/termira-backend.jar
```

第一阶段建议使用 `maven-shade-plugin` 打包 fat jar，方便 Electron Builder 将 Java sidecar 放入应用资源目录。

### 7.3 Java sidecar 启动流程

Electron main 启动时：

1. 找到 Java sidecar 可执行 jar。
2. 创建子进程。
3. 通过 stdio 或本地端口建立通信。
4. 等待 Java sidecar 发送 `backend.ready`。
5. 将后端状态同步给 renderer。

Java sidecar 启动时：

1. 初始化日志。
2. 加载配置目录。
3. 初始化 profile store。
4. 初始化 vault manager。
5. 初始化 session manager。
6. 初始化 sftp manager。
7. 初始化 forwarding manager。
8. 初始化 monitor manager。
9. 发送 ready 事件。

### 7.4 推荐通信方式

第一阶段推荐 `stdio JSON-RPC`。

原因：

- 不占用本地端口。
- 打包后更容易管理。
- Electron main 可以直接管控 Java 进程生命周期。
- 适合桌面应用内部进程通信。

消息格式：

```json
{
  "id": "req_001",
  "type": "request",
  "method": "ssh.connect",
  "params": {}
}
```

响应格式：

```json
{
  "id": "req_001",
  "type": "response",
  "ok": true,
  "result": {}
}
```

事件格式：

```json
{
  "type": "event",
  "event": "terminal.output",
  "payload": {}
}
```

错误格式：

```json
{
  "id": "req_001",
  "type": "response",
  "ok": false,
  "error": {
    "code": "SSH_AUTH_FAILED",
    "message": "Authentication failed",
    "detail": {}
  }
}
```

## 8. IPC 分层

通信链路：

```text
React Renderer
  -> Electron preload
  -> Electron main
  -> Java sidecar
  -> SSH server / local storage
```

Renderer 不能直接访问 Node API，必须通过 preload 暴露受控接口。

Preload 暴露示例：

```ts
window.termira = {
  invoke(method, params),
  on(event, handler),
  off(event, handler)
}
```

Electron main 职责：

- 校验 renderer 调用来源。
- 转发请求给 Java sidecar。
- 管理超时。
- 订阅 sidecar 事件。
- 广播事件给对应窗口。
- 管理 Java sidecar 生命周期。

Java sidecar 职责：

- 执行业务命令。
- 管理 SSH 会话。
- 管理 SFTP。
- 管理端口转发。
- 管理本地配置和 Vault。
- 输出结构化事件。

```text
features/commands/
  CommandPalette.tsx
```

组件原则：

- 业务组件不直接调用 Electron IPC。
- IPC 调用封装在 service 或 store action 中。
- UI 组件接收明确 props。
- 异步状态必须有 loading、success、error。

### 11.2 状态管理

建议拆分 store：

- `useHostStore`
- `useSessionStore`
- `useTerminalStore`
- `useSftpStore`
- `useForwardStore`
- `useMonitorStore`
- `useVaultStore`
- `useUiStore`

运行态数据和持久化数据分离：

- 主机配置来自后端 profile store。
- 会话状态来自 Java sidecar 事件。
- UI 面板展开、当前选中标签等由前端管理。

### 11.4 终端组件

xterm 初始化要求：

- 使用 `@xterm/addon-fit` 自动适配容器。
- resize 需要 debounce。
- terminal input 发送给 `terminal.write`。
- terminal output 通过 `terminal.output` 事件写入。
- 组件卸载时释放 xterm 实例和事件监听。

终端不得：

- 在 React state 中保存完整终端输出。
- 将用户输入写入普通日志。
- 因监控刷新造成输入卡顿。

### 11.5 错误展示

错误展示规则：

- 连接类错误显示在终端标签和状态栏。
- SFTP 错误显示在文件面板内。
- 转发错误显示在规则行内。
- Vault 错误显示在解锁/设置面板。
- 不在 UI 中显示堆栈给普通用户。
- 开发模式可以在 DevTools 中输出脱敏 detail。

### 12.2 Session Manager

`SshSessionManager` 职责：

- 创建 SSH 连接。
- 维护 sessionId -> session 映射。
- 关闭连接。
- 重连。
- 查询状态。
- 向事件总线发送状态变化。

会话状态：

- `CREATED`
- `CONNECTING`
- `AUTHENTICATING`
- `CONNECTED`
- `DISCONNECTING`
- `DISCONNECTED`
- `FAILED`

### 12.3 认证

第一阶段支持：

- password。
- public key file。
- public key content。
- passphrase。
- keyboard-interactive。

认证输入不能写入日志。

认证失败统一返回：

- `SSH_AUTH_FAILED`

连接超时返回：

- `SSH_CONNECT_TIMEOUT`

网络不可达返回：

- `SSH_NETWORK_UNREACHABLE`

### 12.4 终端通道

`TerminalChannel` 职责：

- 打开 PTY shell。
- 读取 stdout/stderr。
- 接收输入。
- 处理 resize。
- 关闭通道。

实现要求：

- 每个终端标签对应一个 channelId。
- channel 输出以事件推给 Electron main。
- 输出事件需要包含 sessionId、channelId、data。
- 输入写入失败时返回明确错误。

### 12.5 SFTP Manager

`SftpManager` 职责：

- 基于 SSH session 创建 SFTP client。
- 列目录。
- 上传。
- 下载。
- 删除。
- 重命名。
- 建目录。

传输任务：

- 每个任务有 transferId。
- 周期性发送进度事件。
- 支持取消。
- 失败后保留错误原因。

### 12.6 Forwarding Manager

支持：

- Local forwarding。
- Remote forwarding。
- Dynamic SOCKS forwarding。

实现要求：

- 每条规则生成 forwardingId。
- 启动前检查端口字段合法性。
- 本地端口占用返回 `FORWARD_PORT_IN_USE`。
- 转发状态变化发送事件。
- 停止时释放资源。

### 12.7 Monitor Manager

监控通过 SSH exec 执行只读命令：

- CPU：`top`、`vmstat` 或 `/proc/stat`。
- 内存：`free` 或 `/proc/meminfo`。
- 磁盘：`df`。
- 网络：`/proc/net/dev`。
- uptime：`uptime` 或 `/proc/uptime`。

要求：

- Linux 优先。
- 命令失败不影响 SSH 主会话。
- 监控频率默认 3 秒。
- 单次采集超时默认 2 秒。

### 12.8 Profile Store

第一阶段推荐 SQLite：

表：

- `host_profiles`
- `host_groups`
- `forward_rules`
- `quick_commands`
- `recent_connections`
- `ui_preferences`

敏感字段不进入 SQLite 明文字段。

### 12.9 Vault Manager

Vault 保存：

- password。
- private key passphrase。
- optional private key content。

Vault 不保存：

- 主机名称。
- host。
- port。
- username。
- 分组。

Vault API 必须只返回调用方需要的凭据，不提供全部解密导出接口。调试模式也不能打印明文。

日志级别：

- `ERROR`
- `WARN`
- `INFO`
- `DEBUG`
- `TRACE`

- `INFO`

- `DEBUG`

## 15. 调试手册

### 15.1 Java sidecar 未启动

检查：

- Electron main 是否找到 jar。
- JDK/JRE 是否可用。
- sidecar stdout/stderr 是否有启动异常。
- `backend.ready` 是否发出。

处理：

- 在 Electron main 增加启动超时。
- 展示后端未就绪错误页。
- 提供重启 sidecar 按钮。

### 15.2 SSH 连接失败

检查：

- host 和 port 是否正确。
- 网络是否可达。
- 认证方式是否正确。
- 私钥格式是否受支持。
- passphrase 是否正确。

日志应包含：

- profileId。
- host 脱敏。
- port。
- 错误码。
- 耗时。

日志不得包含：

- password。
- private key。
- passphrase。

### 15.3 终端无输出

检查：

- SSH session 是否已连接。
- shell channel 是否已打开。
- PTY 是否申请成功。
- Java 是否持续读取输出流。
- IPC 事件是否到达 renderer。
- xterm 实例是否已挂载。

### 15.4 SFTP 卡住

检查：

- SFTP client 是否已创建。
- 当前远程路径是否存在。
- 远端权限是否足够。
- 传输线程是否被阻塞。
- 进度事件是否发送。

### 15.5 端口转发失败

检查：

- 本地端口是否占用。
- 远程目标是否可达。
- SSH 服务端是否允许 remote forwarding。
- dynamic SOCKS 是否绑定正确地址。

### 15.6 Vault 解锁失败

检查：

- 用户是否设置主密码。
- 主密码是否正确。
- Vault 文件是否损坏。
- schema version 是否兼容。

处理：

- 返回 `VAULT_UNLOCK_FAILED`。
- 不透露具体密钥派生细节。
- 不覆盖原 Vault 文件。

## 16. 构建与打包

第一阶段推荐：

- Electron Builder 打包 macOS。
- Java sidecar 使用 Maven 打包为 fat jar。

后续优化：

- 使用 jlink 生成最小 Java runtime。
- 或使用 GraalVM native image 评估体积和启动速度。
